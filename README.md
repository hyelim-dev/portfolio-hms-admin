# HMS Admin Demo Portfolio

실무에서 개발한 교육 운영형 어드민 시스템(HMS)을 기반으로,  
상태 기반 데이터 흐름과 운영 구조를 중심으로 재구성한 프론트엔드 데모 프로젝트입니다.

---

## 📌 Overview

복잡한 전체 기능을 모두 구현하기보다,  
**수강생 → 강의 → 출석 → 통계로 이어지는 핵심 데이터 흐름과 상태 변화**를 중심으로 설계했습니다.

실제 서비스에서는 상담, 수강생관리, 강의관리, 회계관리, 통계, 메시지, 환경설정 등  
다수 도메인이 강하게 연결된 구조였으며,  
데모에서는 이 중 **운영 흐름이 가장 잘 드러나는 영역**을 선별하여 재구성했습니다.

## 🧠 Key Features

### 1. 상태 기반 UI 설계

- 단순 CRUD가 아닌 **상태 변화에 따른 UI 제어 구조**
- 특정 상태에서만 기능이 활성화되도록 설계

예)

- 배정 상태에 따라 수료 버튼 노출
- 상태에 따라 수정 가능 여부 제한

---

### 2. 도메인 간 데이터 흐름 설계

- 수강생 / 강의 / 회계 데이터가 서로 연결된 구조
- 하나의 액션이 여러 영역에 영향을 주도록 설계

예)

- 출석 상태 변경 → 출결 통계 반영
- 환불 처리 → 회계 + 수강 상태 동시 변경

---

### 3. 실제 운영 흐름 기반 구조

- list → detail → 상태 변경 → 데이터 반영 흐름 유지
- 실무 환경과 유사한 UX 구조 설계

## 💡 Demo Strategy

> “조회 흐름은 유지하고, 변경 기능은 제한”

- 핵심 흐름은 실제처럼 동작
- 일부 기능은 데모 환경에 맞게 제한
- mock 데이터 기반으로 구조 유지

---

## 🏗 Architecture Highlights

### 1. 도메인 기반 구조

수강생 → 강의 → 출석 → 통계로 이어지는  
운영형 데이터 흐름 구조를 기반으로 설계했습니다.

---

### 2. 상태 중심 설계

상태값을 기준으로 UI와 기능이 분기되는 구조를 구현했습니다.

- 상태에 따라 버튼 노출/비노출
- 상태 변경 시 후속 데이터 영향 발생

---

### 3. Mock 데이터 전략

- MSW 기반 GraphQL handler 구성
- 도메인별 mock 데이터 구조 통일
- 일부 상태 변경은 localStorage 기반으로 처리

---

## 🔄 Data Flow

실제 서비스에서는 다음과 같은 흐름이 존재했습니다.

- 수강생 상태 변경 → 강의 배정 / 출석부 생성
- 출석 상태 변경 → 출결 통계 반영
- 수강 상태 → 수료율 / 탈락률 계산 반영
- 결제 및 환불 상태 → 회계 데이터 반영

데모에서는 전체 연동을 모두 구현하지 않고,  
**출석 상태 → 출결현황 반영 흐름을 중심으로 재구성했습니다.**

## 🔐 Permission & UI

- 관리자 / 직원 / 강사 권한 분기
- 역할별 기능 및 UI 접근 제어
- 동일 화면에서도 권한에 따라 기능 제한

---

## 🧪 Data Strategy

- 조회 데이터 → MSW 기반 mock
- 상태 변경 → localStorage 기반 mock DB
- 일부 탭 → read-only 유지

---

## 🎯 What I Focused On

- 단순 기능 구현이 아닌 **서비스 흐름 설계**
- 상태 기반 UI 구조와 데이터 연결
- 실제 운영 환경을 고려한 UX 구성

---

## 📝 Retrospective

이 프로젝트는 단순 기능 구현이 아니라,  
실무에서 다루었던 복잡한 운영형 시스템을  
포트폴리오용 데모로 재설계한 작업이었습니다.

모든 기능을 그대로 구현하기보다,

- 무엇을 보여줄지
- 무엇을 제한할지
- 어떻게 흐름을 전달할지

를 기준으로 구조를 다시 설계한 경험이었습니다.
