# HMS Admin Demo Portfolio

실무에서 개발한 교육 운영형 어드민 시스템(HMS Admin)을 기반으로,  
핵심 운영 흐름을 중심으로 재구성한 프론트엔드 데모 프로젝트입니다.

---

## Overview

복잡한 전체 기능을 모두 구현하기보다,  
**강의 → 출석 → 출결현황으로 이어지는 핵심 데이터 흐름과 상태 변화**를 중심으로 설계했습니다.

실제 서비스에서는 상담, 수강생관리, 강의관리, 회계관리, 통계, 메시지, 환경설정 등  
다수 도메인이 강하게 연결된 구조였으며,  
데모에서는 이 중 **운영 흐름이 가장 잘 드러나는 영역만 선별하여 재구성**했습니다.

---

## Key Features

- 강의 상세 → 출석부 진입 흐름 구성
- 출석 상태(출석 / 지각 / 조퇴 / 결석 / 외출) 변경 기능
- 출석 데이터 기반 출결현황 통계 반영
- 상태 기반 UI (배정 여부에 따른 버튼 분기)
- 일부 운영 탭(read-only) 유지

---

## Demo Scope

### Included
- 강의 목록 / 강의 상세 / 출석부
- 출석 상태 변경 및 통계 반영
- 상태 기반 UI 흐름

### Excluded
- 등록 / 수정 / 삭제 전반
- 실제 서버 연동
- 전체 도메인 간 실시간 데이터 연동
- 일부 부가 기능의 최종 저장 / 발송

> 데모에서는 기능 완성도보다  
> **운영 흐름과 구조 전달**에 집중했습니다.

## Architecture Highlights

### 1. 도메인 기반 구조

수강생 → 강의 → 출석 → 통계로 이어지는  
운영형 데이터 흐름 구조를 기반으로 설계했습니다.

---

### 2. 상태 기반 UI 설계

단순 CRUD가 아닌,  
상태 변화에 따라 UI가 분기되는 구조를 구현했습니다.

- 배정 여부에 따라 수료 버튼 노출
- 상태 변경 시 후속 액션 조건 분기
- 실제 운영 환경과 유사한 흐름 유지

---

### 3. Mock 기반 데이터 전략

데모 환경에서는 실제 서버 대신 mock 데이터를 사용했습니다.

- MSW 기반 GraphQL handler 구성
- 도메인별 mock 데이터 구조 통일
- 프론트 요구 shape에 맞춘 데이터 설계
- 일부 상태 변경은 localStorage 기반으로 처리

복잡한 실서비스 구조를 유지하면서도  
데모 환경에 맞게 데이터 흐름을 단순화했습니다.

---

### 4. 성능 최적화

리스트/테이블 중심 구조에서 발생할 수 있는 성능 이슈를 고려했습니다.

- useMemo / useCallback 활용
- lazyQuery 기반 요청 제어
- GraphQL에서 필요한 필드만 요청
- Apollo Client 캐싱 활용
- 페이지네이션(10/20/30/50)
- 검색 범위 제한 (서버 부하 고려)
- Lighthouse 기반 성능 개선

## System Structure (실제 서비스 기준)

본 프로젝트는 실제 HMS Admin 시스템을 기반으로 하며,  
다음과 같은 도메인으로 구성되어 있었습니다.

- 상담관리
- 수강생관리
- 강의관리
- 회계관리
- 통계
- 메시지
- 환경설정

각 도메인은 독립적이지 않고,  
수강생 → 강의 → 출석 → 통계로 이어지는 구조로  
강하게 연결되어 있습니다.

데모에서는 이 중  
**강의 / 출석 중심 흐름을 핵심 시나리오로 선택**했습니다.

---

## Data Flow

실제 서비스에서는 다음과 같은 흐름이 존재했습니다.

- 수강생 상태 변경 → 강의 배정 / 출석부 생성
- 출석 상태 변경 → 출결 통계 반영
- 수강 상태 → 수료율 / 탈락률 계산 반영
- 결제 및 환불 상태 → 회계 데이터 반영

데모에서는 전체 연동을 모두 구현하지 않고,  
**출석 상태 → 출결현황 반영 흐름을 중심으로 재구성**했습니다.

---

## Permission & UI

사용자 역할에 따라 기능과 UI를 분리했습니다.

- 관리자 / 직원 / 강사 권한 분기
- 강사 로그인 시 강의관리만 접근 가능
- 역할별 화면 및 버튼 노출 조건 분리

---

## Data Strategy

데모 환경에서는 다음과 같이 데이터를 구성했습니다.

- 조회 데이터 → MSW 기반 mock
- 상태 변경 → localStorage 기반 mock DB
- 일부 탭 → read-only 유지

---

## Demo Design Strategy

모든 기능을 구현하기보다,  
다음 기준으로 기능을 재구성했습니다.

- 핵심 흐름 → 실제 동작
- 보조 기능 → 데모 안내 후 제한
- 복잡한 영역 → 조회 전용

## Retrospective

이 프로젝트는 단순 기능 구현이 아니라,  
실무에서 다루었던 복잡한 운영형 시스템을  
**포트폴리오용 데모로 재설계한 작업**이었습니다.

모든 기능을 그대로 옮기기보다,

- 무엇을 보여줄지
- 무엇을 제외할지
- 어떻게 흐름을 전달할지

를 기준으로 구조를 다시 설계한 경험이었습니다.
